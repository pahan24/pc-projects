There is no direct or plain SSH mode in this app, all connection to SSH always via local HTTP Proxy Server first even if you set proxy type to none. but we can achieve it using "Response Position" option.

HTTP Response

Response Position

pre_request default value
Sends HTTP Response immediately after SSH connection requested but before payload request. Use this option if you are using plain SSH server and no payload. this is standard HTTP response of SSH with HTTP Proxy connection.
post_request
Sends HTTP Response after payload requested. Replacing any HTTP response status from the server with selected "Response Status" string. Use this option if you are using non plain SSH server (e.g Websocket based server) and use payload.
both
Send HTTP Responses in both positions.

Response Status

Usually you never needed to change this option. default value is a common HTTP. response.

manual
Send HTTP Response using manual value, this option supports payload placeholders.
200 Connection established default value
Sends HTTP Response with hardcoded value "HTTP/1.1 200 Connection established\r\n\r\n".
200 OK
Sends HTTP Response with hardcoded value "HTTP/1.1 200 OK\r\n\r\n".
disabled
Not sending any HTTP response from client side, gets original HTTP response from the server.

HTTP Request (Payload Placeholders)

What are placeholders? These are special keywords that get automatically replaced with actual values when connection requested. They help you create dynamic and flexible HTTP payloads without hardcoding values. Note that these keywords are case sensitive and must in lowercase.

Request Data

These extract HTTP request headers from connection

[method]
Gets replaced with the HTTP method being used in request.
Example: If connection make a GET request, this becomes "GET". in SSH connection always returns "CONNECT"
[protocol]
Gets replaced with the HTTP protocol version being used.
Example: "HTTP/1.1" or "HTTP/1.0"
[host]
Gets replaced with the destination hostname or IP address you're connecting to.
Example: "example.com" or "192.168.1.1"
[port]
Gets replaced with the destination port number you're connecting to.
Example: "443", "22"
[host_port]
Combines the destination host and destination port with a colon separator.
Example: "example.com:443" or "example.com:22"
[proxy_host]
Gets replaced with the proxy address (only if proxy is enabled in your config).
Example: "proxy.example.com"
[proxy_port]
Gets replaced with the proxy address port number (only if proxy is enabled in your config).
Example: "8080" or "3128"
[netdata] or [raw]
Gets replaced with the original request headers without line ending.
Example: "CONNECT example.com:443 HTTP/1.1"
[real_raw]
Gets replaced with the complete original request headers exactly as they were sent.
[auth]
Gets replaced with the proxy authentication header in base64 format (only if proxy authentication is enabled). If not enabled, this becomes empty.
Example: "Proxy-Authorization: Basic am9objpkb2U=" (where the encoded part is your username:password)
[ua]
Gets replaced with the platform specific User-Agent string.

Line Break Characters

These placeholders insert special line break characters.

[cr] or \r
Inserts a Carriage Return character.
[lf] or \n
Inserts a Line Feed character.
[crlf]
Inserts both Carriage Return and Line Feed together (standard line ending in HTTP headers).
[lfcr]
Inserts Line Feed followed by Carriage Return (reverse order).
[cr*x], [lf*x], [crlf*x], [lfcr*x]
Repeats the line break character multiple times. Replace "x" with a number.
Example: [crlf*2] will insert two line breaks, [lf*3] will insert three line feeds

Dynamic Value

These allow you to use different values each time or rotate through values.

[random=value1;value2;value3]
Randomly picks one value from the list each time. Separate values with semicolons.
Example: [random=GET;POST;HEAD] will randomly choose GET, POST, or HEAD each time
[rotate=value1;value2;value3]
Cycles through values in order. First request uses value1, second uses value2, third uses value3, then back to value1.
Example: [rotate=server1;server2;server3] will use server1, then server2, then server3, then repeat

Payload Splitter

These split your request into multiple parts and send them separately.

[split]
Splits your payload at this point and sends the parts as two separate requests immediately
Example: "CONNECT [host]:[port] HTTP/1.1[split]GET / HTTP/1.1[crlf][crlf]..." sends "CONNECT example.com:443 HTTP/1.1" first, then the rest
[instant_split]
Same as [split] but with 0.25 second delay
[delay_split] or [split_delay]
Same as [split] but with 1 second delay
[split=1000]
Splits your payload and sends the second part after a custom delay. Replace the number with milliseconds (1000 = 1 second).
Example: [split=100] waits 0.1, [split=500] waits 0.5 seconds, [split=2000] waits 2 seconds seconds
